## 1. 目标

- 在无串口设备阶段，先完成视觉链路与协议帧输出验证。
- 引入卡尔曼滤波降低中心点抖动与短时丢检影响。

## 3. 当前运行方式
- 探测摄像头索引：
  - python gongxun/camera_index_probe.py --max-index 10
- 启动主识别：
  - python gongxun/disk_color_ring_kalman.py
- 启动 Windows 兼容旧逻辑版本：
  - python gongxun/color_line_det_windows.py
- 运行中快捷键：
  - 0 关闭识别
  - 1~6 切换对应色块/圆环模式
  - q 退出

## 5. 本轮踩坑与解决（2026-03-30）
- 坑 1：摄像头能打开但画面黑帧。
  - 现象：系统相机设置能预览，但程序窗口黑屏或取流异常。
  - 根因：过度参数协商（强制 MJPG + 多分辨率循环 + 反复 set）导致驱动协商失败。
  - 解决：改用稳定初始化（固定参数、单次设置、短预热），移除高风险协商逻辑。
- 坑 2：程序误选内置摄像头。
  - 现象：外接相机已接入，但程序实际抓取到笔记本内置镜头。
  - 根因：index 回退策略过宽。
  - 解决：新增索引探测工具并确认 index=2，主程序固定优先索引为 [2]，默认禁用 0 号回退。
- 坑 3：无串口设备阶段无法验证输出链路。
  - 现象：识别逻辑运行但无法看到协议发送结果。
  - 解决：发送链路改为打印协议帧（55 5A action x y AA），保留 serial 模式开关。
- 坑 4：用户不希望继续改原文件，避免反复撤销影响节奏。
  - 现象：对原文件直接改动容易与现场验证版本冲突。
  - 解决：改为新建独立文件 color_line_det_windows.py，原文件保持不动。
- 坑 5：HF899/HF901 在树莓派下不强制 MJPG 时易黑屏。
  - 现象：设备可打开但画面黑屏或假图，尤其在 Linux/V4L2 下更明显。
  - 根因：默认像素格式不确定，落到 YUY2/YUYV 后解码性能与兼容性不稳定。
  - 解决：Linux 分支强制 MJPG + 固定 640x480 + /dev/videoX 扫描与帧均值验收。
- 坑 6：Windows 下同样出现黑屏，厂商建议走 UVC 方式调用。
  - 现象：在 Windows 上摄像头可被识别，但取流窗口黑屏。
  - 根因：后端/像素格式协商不稳定，默认格式可能不是该设备稳定输出格式。
  - 解决：Windows 分支改为 UVC 友好策略：
    - 后端依次尝试 DSHOW -> MSMF -> CAP_ANY
    - 像素格式依次尝试 MJPG -> YUY2 -> default
    - 每组组合预热后多帧探测，拿到非全黑帧才确认启用
- 坑 7：Windows 探测失败时回退到了 0 号内置摄像头。
  - 现象：外接相机暂时不可用时，程序自动切到电脑自带摄像头。
  - 根因：候选索引列表默认包含 0 号回退。
  - 解决：新增配置并默认关闭回退：
    - WINDOWS_CAMERA_INDICES = [2]
    - ALLOW_WINDOWS_INDEX_0_FALLBACK = False
    - 找不到外接相机时直接报错并退出，不再误开内置摄像头
- 坑 8：Windows 出现“一半花屏一半黑屏”。
  - 现象：画面非纯黑，但存在明显半屏异常（一侧花屏/一侧发黑）。
  - 根因：UVC 后端与像素格式组合不匹配，单帧可读不代表画面可用。
  - 解决：
    - Windows 端改为优先 YUY2，再尝试 MJPG/default。
    - 增加帧质量筛选：拒绝 too-dark / mostly-black / half-black。
    - 仅在通过质量检查后确认相机组合可用。
- 坑 9：DSHOW 在当前设备上返回全暗帧，但 CAP_ANY 可正常取流。
  - 现象：日志显示 backend=700 下 YUY2/MJPG/default 全部 too-dark；backend=0 + YUY2 正常。
  - 根因：当前摄像头驱动与 DSHOW 协商不稳定。
  - 解决：Windows 探测顺序调整为优先 CAP_ANY，再回退 DSHOW/MSMF。
- 坑 10：当前现场已确认 HF899 实际在 index=0，旧索引配置会误导探测。
  - 现象：按旧配置优先探测 index=2，导致反复尝试不稳定组合。
  - 根因：设备索引发生变化，但程序未同步更新。
  - 解决：
    - CAMERA_INDEX 改为 0
    - WINDOWS_CAMERA_INDICES 改为 [0]
    - 调试阶段放宽亮度门槛，避免低照度被误判
    - 分辨率切换到 1920x1080 以匹配 HF899 原生高分辨率能力
- 坑 11：用户需要对外讲解代码，但色环流程与色块流程混在一起不直观。
  - 现象：模式 4/5/6 看起来像普通色块阈值，不利于讲解和后续调参。
  - 根因：没有独立的色环检测函数和独立分支。
  - 解决：新增 find_ring 与独立模式分支，显式区分色块识别和色环识别。
